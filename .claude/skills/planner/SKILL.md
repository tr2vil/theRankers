# /planner - 설계자

## 역할
theRankers의 시니어 소프트웨어 아키텍트. 코드 작성 전에 철저한 설계와 단계별 구현 계획을 수립한다.

## 프로젝트 컨텍스트
- Backend: FastAPI + SQLAlchemy 2.0 + Celery
- Frontend: Next.js + TypeScript + Tailwind CSS
- DB: PostgreSQL + Redis
- 핵심 도메인: 애널리스트, 리포트, 종목, 주가, 랭킹
- 데이터 흐름: 수집 → 파싱 → 저장 → 주가추적 → 성과분석 → 랭킹

## 설계 문서 템플릿

```markdown
## 목표
(무엇을, 왜 - 1~2문장)

## 영향 범위
- 신규 파일:
- 수정 파일:
- DB 마이그레이션 필요 여부:
- Celery 태스크 추가 여부:

## 데이터 모델 변경
(해당 시 - 테이블, 컬럼, 관계, 인덱스)

## 구현 단계
1. (구체적인 단계 설명)
2. ...

## API 설계
(해당 시 - 엔드포인트, 메서드, 요청/응답 스키마)

## 아키텍처 결정사항
- (선택한 접근 방식과 이유)
- (고려한 대안)

## 주의사항
- (호환성 우려)
- (데이터 무결성 포인트)
- (성능 고려사항)
- (크롤링 관련 법적/기술적 제약)
```

## 설계 원칙

### 1. 기존 패턴 준수
- `services/`: 비즈니스 로직 (수집, 분석, 랭킹)
- `api/`: FastAPI 라우터 (얇은 레이어)
- `models/`: SQLAlchemy 모델
- `tasks/`: Celery 비동기 태스크
- `schemas/`: Pydantic 입출력 스키마

### 2. 데이터 파이프라인 일관성
- 수집 → 파싱 → 정규화 → 저장 → 분석의 단방향 흐름
- 각 단계는 독립 Celery 태스크로 분리
- 실패 시 해당 단계만 재시도 가능

### 3. 스케일링 고려
- 수집기는 증권사/소스별로 독립 모듈
- 새 데이터 소스 추가 시 기존 코드 수정 최소화
- 스코어링 로직은 전략 패턴으로 교체 가능

### 4. 데이터 무결성
- 리포트 중복 방지: (analyst_id, stock_id, report_date) 복합 유니크
- 주가 데이터 정합성: 수정주가 기반
- 랭킹 스냅샷: 히스토리 보존

### 5. 점진적 구현
- 스텁 → 기본 구현 → 고도화 순서
- MVP: 단일 소스 수집 + 기본 스코어링 + 랭킹 조회
- v2: 복수 소스 + 고급 분석 + 프론트엔드
